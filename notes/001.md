# Development Workflow

---

# Phase 1 - Understand

Before editing:

Analyze:

* Existing files.
* Current architecture.
* Existing behavior.
* Technical limitations.
* Potential risks.

Do not immediately modify files.

First explain:

* What exists.
* What works.
* What needs improvement.
* What risks exist.

---

# Phase 2 - Plan

Create an implementation plan.

Include:

* Files that will change.
* New files required.
* Architecture decisions.
* Implementation approach.
* Testing strategy.

The plan must respect:

* Existing project structure.
* Coding standards.
* Technology stack.
* Future roadmap.

---

# Phase 3 - Implement

Modify the code.

Requirements:

* Keep existing functionality working.
* Prefer clean solutions.
* Improve maintainability.
* Follow professional engineering practices.

Prefer:

* Small controlled changes.
* Clear structure.
* Reusable logic.
* Good naming.

Avoid:

* Random dependencies.
* Unnecessary frameworks.
* Complete rewrites without justification.
* Duplicate logic.
* Breaking existing features.

# Milestone Review Requirement

After completing each roadmap task:

The developer must review:

* git diff
* application behavior
* documentation updates

before starting the next task.

The completed task must be committed to Git before beginning the next roadmap item.

Required workflow:

1. Complete assigned task.
2. Review modified files.
3. Test application behavior.
4. Verify DEVELOPMENT_STATUS.md.
5. Verify CHANGELOG.md.
6. Commit changes.
7. Begin next roadmap task.

Do not begin the next task until the previous task has been committed and the working tree is clean.
